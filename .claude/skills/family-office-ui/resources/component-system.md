# Component system

## Shell
The shell should include:
- slim top bar or left rail with understated branding
- mode-aware navigation grouped as Family, Operations, Advisory
- large page title area with optional context line
- one primary action per page max

Avoid loud app chrome.
The shell should disappear into the background once the content loads.

## Panels
Use three panel types only.

### 1. Salon panel
For family content.
- larger padding
- softer radius
- image-forward
- generous title/caption spacing
- supports quote, story, gallery, or featured profile

### 2. Ledger panel
For operational content.
- medium padding
- stronger border/grid logic
- supports lists, status rows, maps, and schedule summaries
- built for scanning and management

### 3. Briefing panel
For advisory content.
- tighter but elegant spacing
- subtle rules and sections
- supports memo summaries, document metadata, key figures, and recommendation blocks

## Cards
Cards are secondary, not the whole layout.
Rules:
- cards should usually appear in mixed scales, not perfectly repeated rows
- one large anchor card is better than six small equal cards
- avoid decorative icons unless functionally useful
- use eyebrow text and secondary metadata to create hierarchy

## Tables and lists
For property and advisory pages, prefer refined tables/lists with:
- generous row height
- faint dividers
- sticky headers where useful
- inline status pills that are subdued, not candy-colored
- typography doing most of the work

## Photos
Photos should not be trapped in tiny tiles by default.
Use:
- featured image with supporting strip
- masonry only if content is inherently irregular
- filmstrip or journal layouts for events and memories
- metadata and captions beneath or alongside images

## Profile modules
Profiles should feel authored, not CRM-like.
Include:
- strong portrait or monogram
- role in family or context line
- key relationships or interests
- recent updates or contributions
- linked collections like homes, photos, events, or documents

## Property modules
A property view should combine:
- hero image or map
- quick facts block
- active tasks/vendors
- recent expenses or documents
- seasonal reminders or upkeep timeline

## Advisory modules
An advisory page should combine:
- succinct top summary
- key figures in restrained statistic style
- documents/recommendations below
- named advisors with role and recency
- action log or important dates in a memo tone

## Buttons and controls
- Primary button: dark ink or deep accent, solid fill
- Secondary button: quiet surface with border
- Tertiary control: text link or ghost style
- Keep button copy short and literal

## Search and filters
Make filters feel tailored, not enterprise-heavy.
Use segmented controls, lightweight filter chips, or a compact filter tray.
Avoid giant dense filter bars.
